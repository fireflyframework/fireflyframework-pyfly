Chapter 6

Commands & Queries: CQRS for Transfers

By now the wallet can hold money, but every change still runs through one fat service. In this chapter we split writes from reads using PyFly's CommandBus and QueryBus, so a money transfer becomes a small, testable, independently-authorized unit of work.

Modelling a transfer as a command

A command is an intention to change state. We declare it as a typed message and let the bus route it to exactly one handler:

wallet/core/transfers/transfer_funds.py 
from pyfly.cqrs import Command, CommandHandler, command_handler
from pyfly.container import service

class TransferFunds(Command[TransferResult]):
    from_wallet: str
    to_wallet: str
    amount: Money

@command_handler
class TransferFundsHandler(CommandHandler[TransferFunds, TransferResult]):
    def __init__(self, ledger: LedgerRepository, events: EventPublisher) -> None:
        self._ledger = ledger
        self._events = events

    async def handle(self, cmd: TransferFunds) -> TransferResult:
        # post a double-entry: debit source, credit destination
        entry = await self._ledger.post(cmd.from_wallet, cmd.to_wallet, cmd.amount)
        await self._events.publish(FundsTransferred(entry.id))
        return TransferResult(entry_id=entry.id)
Listing 6.2 — A command and its single handler, wired by the bus.
NoteThe bus resolves the handler by the command's type — no manual registration. One command, one handler, enforced at startup.
Spring parityPyFly's @command_handler mirrors Spring's @CommandHandler + Axon bus — same dispatch model, expressed with Python type hints.
